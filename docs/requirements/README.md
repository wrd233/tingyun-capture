# tingyun-capture 需求文档包

当前 v2 增量合同见 `v2-capture-research-workstation.md`。它覆盖 v1 的 Greenfield 假设：v2 必须兼容读取 v1 Session 与 AI-ready，不原地迁移或重写旧 Raw。

> 状态：需求已完成深度 grill-me 收口，可进入设计与实现阶段。  
> 范围：全新 Greenfield 重建；macOS；本地单用户；纯代码；零 LLM。  
> 目标：把用户在真实听云页面中的需求驱动探索，记录成高保真、可审阅、AI 可直接消费的证据包。

## 1. 一句话定义

`tingyun-capture` 不是抓包器，也不是自动化探索 Agent，而是一个 **需求驱动的真实平台探索记录器**：

```text
用户真实探索听云页面
        ↓
tingyun-capture 纯代码忠实记录
        ↓
Raw Evidence + 人工说明
        ↓
纯代码脱敏与整理
        ↓
AI-ready Evidence Package
        ↓
外部 Claude Code / Codex / Agent 分析
```

四条不可动摇的边界：

1. 用户负责真实探索。
2. Capture 负责纯代码忠实记录。
3. 外部 AI 负责分析、推理、Endpoint 发现、参数血缘和 Skill 化。
4. 未来其他工具负责 Replay、自动遍历和自动化执行。

## 4. 需求优先级

### P0：必须正确

- 单 Session / 单 Step 的事实边界与状态机。
- 多 Tab 自动纳入当前 Session。
- 目标 Origin 内的高保真网络记录。
- Request / Response / Body 增量落盘。
- 关键页面操作、表单状态、选项与真实提交记录。
- Raw 与 AI-ready 严格分层。
- 崩溃、浏览器关闭、Engine 停止、磁盘和写入失败时不伪装完整。
- AI-ready 有唯一入口、稳定引用和完整性说明。

### P1：必须好用

- 清爽的 Sidecar。
- Active Step 的少量实时高价值事件反馈。
- Session Review 轻量下钻到 Request / Response。
- 时间点备注。
- 参考 cURL。
- AI-ready ZIP。

### P2：不阻塞 v1

- 更丰富的性能 Timing。
- 特殊资源类型的额外兼容。
- 更深的测试夹具覆盖。

## 5. 明确未在需求阶段锁定的实现细节

下列内容应在设计阶段决定，不应被误写成已经确认的产品要求：

- 后端语言与 Web 框架。
- 前端框架。
- 浏览器控制层最终采用 Playwright、CDP 直连或组合方式。
- 具体单 Body 默认大小上限。
- 磁盘“低空间 / 危急空间”的具体阈值。
- 端口号。
- Raw 目录的最终物理命名，只要逻辑分层满足本契约。

但无论采用何种实现，必须遵守本文档包中的行为契约。
