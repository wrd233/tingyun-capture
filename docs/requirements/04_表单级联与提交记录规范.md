# 04 表单级联与提交记录规范

## 1. 目标

表单能力是 v1 的核心价值之一。

目标不是做表单自动化，而是忠实回答：

1. 用户当时看到哪些选项。
2. 用户实际选了什么。
3. 上游操作后，下游表单状态实际怎样变化。
4. 最终提交前，页面处于什么状态。
5. 用户真实提交后，哪些请求和响应发生在附近。

## 2. 总原则

```text
用户负责探索
Capture 负责观察
```

Capture 永远不：

- 主动打开下拉框。
- 主动切换选项。
- 自动遍历组合。
- 自动提交表单。

因此“所有选项”的准确含义是：

> 本次真实探索过程中，已经存在、已经加载或已经展示出来的全部选项。

## 3. 控件类型

### 3.1 Input

记录：

- 字段语义上下文。
- 初始值（能可靠获得时）。
- 最终值。
- 修改时间。

不记录逐键过程。

### 3.2 Select / 自定义下拉

尽可能记录：

- 当前字段名称。
- 当前已暴露的选项集合。
- 当前选中项。
- 选项显示文字。
- 页面明确暴露的技术值。

### 3.3 Radio / Checkbox

记录：

- 当前可选项。
- 显示文字。
- 技术值（能确定时）。
- 选中状态。

### 3.4 Enabled / Disabled / Visible State

当与级联相关时，记录：

- disabled → enabled。
- enabled → disabled。
- visible → hidden。
- hidden → visible。

只记录确定性状态变化。

## 4. 显示文字与技术值

### 4.1 两者都保留

例如：

```text
label: OA 系统
value: 2033
```

如果页面明确暴露，两者都保存。

### 4.2 拿不到不猜

不允许：

- 从附近请求自动推断 option value。
- 猜测某个数字就是 applicationId。
- 深入解析 Angular / React 私有对象来强行抽取。

页面事实与网络事实分别保存。

外部 AI 后续判断两者关系。

## 5. 级联观察模型

### 5.1 不持续监听整个 DOM

不记录所有：

- 元素增删。
- class 变化。
- 动画。
- 无关区域重绘。

### 5.2 围绕真实操作观察

用户操作关键表单控件时：

```text
操作前
→ 保存相关表单状态

用户真实操作
→ 记录操作与网络

操作后
→ 在有限窗口内观察相关表单区域
→ 状态稳定或窗口结束后保存新状态
→ 生成前后差异事实
```

### 5.3 相关区域优先级

优先：

1. 当前控件所属真实 `<form>`。
2. 可确定的附近表单容器。
3. 页面中已经识别出的相关表单控件集合。

不需要用户手工选择区域。

### 5.4 记录事实，不写因果

允许写：

```text
15:03:21 用户选择“应用 = OA”
15:03:21.120 发起请求 X
15:03:21.700 事务选项集合由 [] 变为 [登录, 查询, 保存]
```

不允许 Capture 写：

```text
选择 OA 导致事务列表变化
```

因果由外部 AI 判断。

## 6. 实际探索分支

Capture 只记录用户真正走过的分支。

例如用户操作：

```text
应用 A
→ 事务 1

应用 B
→ 事务 4
```

Capture 保存这两条真实路径。

未选择的应用 C 以及其下级选项不应被凭空补齐。

## 7. 提交触发识别

### 7.1 目标

在用户真实执行保存 / 提交前，自动保存一次相关表单完整状态。

### 7.2 纯代码高置信度规则

可使用：

#### 标准语义

- form submit event
- 用户点击可见、可用、且关联真实 `<form>` 的 `button[type=submit]`
- 用户点击可见、可用、且关联真实 `<form>` 的 `input[type=submit]`
- 表单字段内的 Enter 提交

#### 禁止的误判来源

- 页面上存在隐藏 submit button。
- 普通 `<a>` 导航点击。
- 普通菜单项文字包含“应用”“确定”等词。
- 未关联 form 的 role/button 文本。

### 7.3 触发信息

有效 `submit_window_opened` 必须记录可审计 trigger，例如：

```text
trigger.kind
trigger.interaction_id（能关联时）
trigger.form_id / form_name（能确定时）
```

Capture 绝不因为识别触发而改变页面。

## 8. 提交前完整表单状态

### 8.1 触发时机

必须发生在用户真实点击提交候选之前。

### 8.2 内容

尽可能记录相关表单的完整当前状态：

- 字段名称。
- 控件类型。
- 当前值。
- 当前选中项。
- 当前已暴露选项集合。
- 技术值（页面明确暴露时）。
- checked 状态。
- enabled / disabled。

### 8.3 独立事实

提交前状态拥有稳定 ID，例如：

```text
form-state-0017
```

不覆盖此前的操作历史。

## 9. 提交操作与网络事实

证据链应保留：

```text
历史表单变化
        ↓
提交前完整表单状态
        ↓
用户真实点击提交
        ↓
附近真实 Request / Response
```

Capture 不自动判断：

- 哪个请求是最终 save 接口。
- 哪个字段对应哪个 Payload 参数。
- 哪个 Response 表示业务成功。

## 10. 提交结果观察窗口

### 10.1 仅是轻量时间索引

提交候选发生后，建立默认约 5 秒的观察窗口。

它不新增采集行为，只索引本来就会记录的事实。

### 10.2 可包含

- 新 Request / Response。
- 失败请求。
- URL 变化。
- Tab 变化。
- 可确定读取的表单状态变化。

### 10.3 不做

- 等待网络完全静止。
- 成功 / 失败业务判断。
- 自动找真正保存接口。
- 复杂请求关联图。
- 多阶段状态机。

### 10.4 语言边界

Capture 只能表达：

> 这些事实发生在提交后的观察窗口内。

不能表达：

> 这些事实由提交操作导致。

## 11. 敏感输入

交互日志中的高置信度凭据值不得保存。

例如：

- 密码。
- 明确 Token 输入框。
- 高置信度 Secret 字段。

只记录：

```text
field changed = true
value = ***NOT_CAPTURED***
```

这样避免在页面操作层重复复制秘密。

## 12. 典型证据示例

```text
Step: 探索应用告警配置级联

interaction-001
选择“应用 = OA”

form-state-before-001
事务 options = []

request-0042
GET /transactions?applicationId=2033

form-state-after-001
事务 options = [登录, 查询, 保存]

interaction-0088
填写“严重阈值 = 5000”

form-state-0017
提交前完整状态

interaction-0091
点击“保存”

submit-window-0003
后续约 5 秒客观事实索引

request-0102
POST /.../save

response-0102
200
```

Capture 到此为止。

外部 AI 再判断：

- 哪个 Endpoint 最关键。
- `applicationId=2033` 与 OA 的关系。
- `5000` 映射到哪个 Payload 字段。
- 是否可以直接通过 API 创建相同配置。
