# 01 产品需求与范围

## 1. v1 产品边界

v1 是：

> 高保真记录器 + Session / Step 组织 + 简单 Review + AI-ready 证据包。

v1 对“事实是否被忠实保存”负责。

v1 不对以下结论正确性负责：

- 哪个 Endpoint 最重要。
- 哪个参数来自哪个前置请求。
- 哪次操作导致哪次请求。
- 哪个 URL 是稳定深链。
- 哪条路径应该固化成 Skill。

## 2. 核心使用场景

### 场景 A：完整记录一次需求驱动探索

用户围绕一个主题开始 Session，例如：

- 告警到 Trace。
- 应用告警配置探索。
- 事务详情页面接口探索。

过程中：

- 多个 Tab 自动纳入。
- 用户开始和结束多个 Step。
- Step 之间仍持续记录。
- 用户可以随手写时间点备注。
- Request / Response 持续落盘。

### 场景 B：研究复杂表单和级联

用户真实操作：

```text
选择应用
→ 下游事务选项变化
→ 选择事务
→ 指标变化
→ 填写阈值
→ 点击保存
```

Capture 应记录：

- 实际暴露出来的选项集合。
- 显示文字。
- 页面明确暴露的技术值。
- 操作前后表单状态变化。
- 提交前完整表单状态。
- 真实提交 Request / Response。

### 场景 C：问题到 Trace 和 URL

用户从告警或问题出发：

```text
应用
→ 事务
→ Trace
→ 具体页面
```

Capture 应保留：

- 关键页面操作。
- 新 Tab。
- 所有 URL 变化，包括 SPA。
- 业务请求与响应。
- 过程中出现的 ID。
- 用户备注时的 URL 上下文。

### 场景 D：长时间可靠探索

一次 Session 可以持续较长时间：

- 大量请求。
- 多 Step。
- Step 之间停顿。
- Sidecar 刷新或关闭后重开。
- 请求完成晚于 Step 或 Session 边界。

数据必须增量落盘，不能等结束时一次保存。

## 3. 核心能力清单

### 3.1 Session

- 全局仅允许一个 Active Session。
- Session 是连续高保真录制边界。
- 开始 Session 只要求用户填写名称。
- 允许开始时没有任何目标 Origin 页面。
- 正常结束后进入 FINALIZING，再 SEALED。
- 不支持 Pause。

### 3.2 Step

- 一个 Session 内仅允许一个 Active Step。
- 不支持嵌套和并发。
- 开始 Step 必须填写一句短的“操作意图”。
- Step 结束后可填写或后补结果 / 观察。
- Step 边界一旦发生，事后不可修改。
- Step 不控制采集开关。

### 3.3 时间点备注

- Session Active 期间始终可用。
- 有 Active Step 时自动关联当前 Step。
- 无 Active Step 时只关联 Session。
- 自动绑定：时间、活动 Tab、当前 URL、页面标题。
- 用户只需写一句话。

### 3.4 多 Tab

- Dedicated Chromium 中所有目标 Origin Tab 自动纳入 Active Session。
- 一个 Step 可以跨多个 Tab。
- Tab 需要稳定 ID。
- 记录创建、来源、URL、标题、激活切换、关闭。

### 3.5 网络

- 目标 Origin 内每个网络事件至少留下元数据索引。
- 动态 / 业务请求保存完整 Request / Response 内容。
- 明显静态资源默认只保存元数据。
- 不以 API 前缀作为采集底层边界。

### 3.6 页面操作

只记录关键离散操作：

- Click。
- Input 最终值。
- Select。
- Checkbox / Radio。
- Enter / Submit。
- Tab create / switch / close。
- Navigation / URL change。

不记录：

- 鼠标轨迹。
- 滚动轨迹。
- 每次按键。
- 完整 DOM。

### 3.7 表单

- 记录用户实际探索时已经暴露的选项。
- 不自动遍历。
- 记录显示文字与可确定技术值。
- 操作前后做有限表单状态观察。
- 提交候选前保存完整表单状态。
- 提交后建立轻量观察窗口。

### 3.8 Review

- 清爽、三级下钻。
- 默认查看 Raw。
- 支持极简 URL / Path 搜索。
- 支持失败请求固定筛选。
- 支持查看 Request / Response。
- 支持复制参考 cURL。

### 3.9 AI-ready

- Capture 内部零 LLM。
- 正常封存时自动生成。
- 有唯一 `README_FOR_AI.md` 入口。
- 有机器可读 JSON / JSONL 索引。
- 有证据 ID 引用。
- 有完整性摘要。
- 可重新生成。
- 可打包为不含 Raw 的 ZIP。

## 4. 作用域

### 4.1 目标 Origin

正常使用时唯一必需配置：

```yaml
target_origin: <scheme://host:port>
```

规则：

- 只对该 Origin 做完整采集。
- 不做动态跨 Origin 发现。
- Tab 离开目标 Origin 后，外部内容完全忽略。
- 返回目标 Origin 后恢复采集。

### 4.2 无 API 前缀白名单

不能把以下前缀写成采集底层边界：

- `/nalarm-api`
- `/server-config`
- `/platform-api`
- `/server-api`
- 其他已知前缀

原因：

> 用户需求可能跨多个前缀，甚至出现此前未知的新接口。

## 5. v1 的简化原则

### 5.1 不因为少数边缘场景增加新产品概念

例如：

- 不为 Step 之间的时间增加用户可见“between_steps”概念。
- 不为 iframe 建独立工作区。
- 不为外部 Origin 建跨域追踪模型。
- 不为两次 Session 建对比系统。

### 5.2 允许“事实丰富、界面克制”

底层可以保存较完整事实，但 UI 只突出重要内容。

例如网络 Timing：

- 基础时间必须保存。
- 底层直接可得的更细 Timing 可以附带保存。
- Sidecar 和 Review 默认不突出。

## 6. 配置需求

### 6.1 普通用户需要知道的

- `target_origin`

### 6.2 有默认值的

- 输出目录。
- Browser Profile 目录。
- 单 Body 大小上限。
- Finalization timeout，默认 10 秒。
- 提交结果观察窗口，默认约 5 秒。
- 网络分类策略。
- 默认脱敏规则。
- 最近事件展示数量。

### 6.3 少量高级覆盖

允许通过配置文件覆盖少数参数，例如：

- 输出目录。
- Profile 目录。
- Body 上限。
- 额外敏感字段名。

不建设复杂设置页面。

## 7. 平台与兼容性范围

### 7.1 Greenfield

新版彻底从零开始：

- 删除旧 Capture 内容。
- 不导入旧产物。
- 不迁移旧 schema。
- 不兼容旧目录。
- 不复用旧实现作为约束。

旧版只作为需求经验和测试参考。

### 7.2 平台

仅 macOS 正式支持和验收。
