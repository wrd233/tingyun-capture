# 11 实现交付清单与阅读顺序

> 本文不是详细技术设计，而是把已经确认的需求转成实现阶段的交付边界。  
> 具体框架和代码结构应另写设计文档，不应反向修改本需求契约。

## 1. 实现 Agent 必须先读

顺序：

```text
README.md
→ 00 项目定位
→ 01 产品范围
→ 07 生命周期与可靠性
→ 03 采集引擎
→ 04 表单与提交
→ 05 Raw 契约
→ 06 AI-ready 契约
→ 02 Sidecar
→ 08 测试
→ 09 非目标
→ 10 决策台账
```

原因：

先理解边界和状态机，再做采集与 UI，避免从前端页面开始堆功能。

## 2. 建议实现阶段

### 阶段 1：Greenfield 骨架

必须交付：

- 一条命令启动。
- Engine。
- Sidecar 空壳。
- Dedicated Chromium + persistent profile。
- 127.0.0.1 only。
- 配置只要求 target_origin。

完成定义：

- start 能拉起三部分。
- stop 能关闭 Chromium。
- 不包含旧 Capture 兼容代码。

### 阶段 2：Session / Step / Note 状态机

必须交付：

- IDLE / ACTIVE / FINALIZING / SEALED / INTERRUPTED。
- 全局单 Session。
- 单 Active Step。
- Step 意图必填。
- 时间点备注始终可用。
- Sidecar 关闭不影响 Engine。

完成定义：

- 所有边界行为有自动化测试。

### 阶段 3：Raw 增量持久化

必须交付：

- manifest。
- append-only events.jsonl。
- stable IDs。
- 独立 Body 文件。
- 基础完整性缺口记录。
- Crash 后发现 INTERRUPTED。

完成定义：

- 长 Session 不依赖内存保存全部数据。
- 关键写失败能中断。

### 阶段 4：Tab / Frame / URL / Interaction

必须交付：

- 多 Tab 自动纳入。
- Tab 生命周期。
- Frame 轻量上下文。
- SPA URL 变化。
- 关键页面操作。
- 语义控件快照。
- Session Start Baseline。

完成定义：

- 本地测试站点能端到端验证。

### 阶段 5：网络高保真

必须交付：

- Request 发起即落盘。
- 生命周期终态。
- 全 Headers。
- Body。
- Body 上限。
- Redirect chain。
- Initiator。
- 基础 Timing。
- Cache facts（可得时）。
- WebSocket/EventSource 轻量事实。
- Download。

完成定义：

- Step / Session 边界 pending request 用例通过。

### 阶段 6：表单级联与提交

必须交付：

- Input final value。
- 选项集合。
- label + technical value。
- 操作前后状态。
- 提交候选识别。
- 提交前完整表单状态。
- 约 5 秒轻量结果窗口。

完成定义：

- 本地级联与 save 测试端到端通过。

### 阶段 7：可靠性

必须交付：

- 10 秒 finalization。
- Browser close → Interrupted。
- Engine stop → Interrupted。
- 低磁盘警告。
- 危急磁盘中断。
- persistence failure 中断。
- Interrupted 阻止新 Session。

### 阶段 8：Review

必须交付：

- 最近 Session 列表。
- 清爽三级下钻。
- Raw 默认展示。
- 极简 URL/Path 搜索。
- 全部/失败筛选。
- Request/Response 查看。
- 参考 cURL。
- AI-ready 状态与重新生成。
- 删除整个 Session。

不允许实现成复杂分析台。

### 阶段 9：AI-ready

必须交付：

- Deterministic `README_FOR_AI.md`。
- session.json。
- events.jsonl。
- network-index.jsonl。
- evidence request/response。
- 宽松高置信度脱敏。
- 完整性摘要。
- READY / FAILED / STALE。
- AI-ready ZIP。
- 二进制排除规则。

### 阶段 10：测试与真实验证准备

必须交付：

- 小型本地测试站点。
- 端到端自动化测试。
- 三类真实人工验证指南。
- Fresh AI 阅读测试指南。

## 3. 完成定义（Definition of Done）

v1 不能只以“页面能打开”作为完成。

至少满足：

### 3.1 功能

- Session / Step / Note 全流程可用。
- 多 Tab 可用。
- 网络事实可靠落盘。
- 表单级联可记录。
- 真实提交证据链完整。
- Review 清爽可用。
- AI-ready 可生成和打包。

### 3.2 可靠性

- Crash 不丢全部 Session。
- Browser close 诚实中断。
- Engine stop 诚实中断。
- Body 超限明确 gap。
- 写入失败不继续假装正常。

### 3.3 安全

- 本地服务只监听 localhost。
- Raw 默认私有。
- AI-ready 有脱敏。
- ZIP 不含 Raw。
- 二进制不进入 AI-ready。

### 3.4 文档

实现后必须新增设计文档，至少说明：

- 代码模块关系。
- 状态机实现。
- Raw schema。
- AI-ready schema。
- 故障处理。
- 本地运行方式。
- 测试方式。

设计文档不得取代本需求包。

## 4. 实现过程中禁止自行扩张

如果实现 Agent 想加入以下内容，应停止并回看非目标：

- AI 分析按钮。
- 自动 Endpoint 排序。
- Replay。
- 自动表单遍历。
- Screenshot。
- DOM snapshot。
- 完整 Network 面板。
- Session Diff。
- Pause。
- 多 Session 并发。
- 登录系统。
- 桌面 App。

## 5. 建议的交付检查问题

每轮实现结束至少回答：

1. 这轮新增了哪些用户可见概念？是否必要？
2. 是否有任何 Capture 主动操作真实页面？
3. 是否有任何 LLM 依赖？
4. 是否有事实只存在内存、未增量落盘？
5. Raw 与 Derived 是否仍然清晰分离？
6. 是否有失败被静默吞掉？
7. UI 是否仍然清爽？
8. 是否加入了非目标能力？

## 6. 最终交付物建议

```text
源码
+

+
设计文档
+

+

```
