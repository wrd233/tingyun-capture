# 08 测试与人工验证计划

## 1. 测试策略

采用两层：

```text
自动化测试
→ 稳定验证确定性行为和关键端到端链路

真实听云人工验证
→ 由用户决定何时操作和是否符合真实需求
```

不要求 CI 自动访问真实听云。

## 2. 自动化测试分类

### 2.1 单元测试

重点覆盖：

- Session 状态机。
- Step 状态机。
- Request 归属。
- Finalization 规则。
- Stable ID。
- JSONL 持久化。
- Body 文件引用。
- Redaction。
- cURL 生成。
- AI-ready 生成。
- 完整性摘要。

### 2.2 故障测试

覆盖：

- Browser 关闭。
- Engine 停止。
- Capture 崩溃后恢复。
- 关键持久化失败。
- Body 超限。
- 磁盘低空间 / 危急空间。
- AI-ready 生成失败。

### 2.3 本地端到端测试站点

仅测试时使用。

不是产品模块，不模拟完整听云。

建议提供少量固定页面：

#### 页面 1：普通请求 + 新 Tab

验证：

- Tab 自动纳入。
- opener 关系。
- URL 变化。
- 请求落盘。

#### 页面 2：简单级联表单

例如：

```text
应用 A / B
→ 异步加载下级选项
```

验证：

- 选项集合。
- 显示文字。
- 技术值。
- 前后状态。
- 相关网络事实。

#### 页面 3：最终提交

```text
填写字段
→ 点击保存
→ POST /save
→ 固定 JSON Response
```

验证：

- 提交候选识别。
- 提交前表单状态。
- 提交观察窗口。
- Request / Response Body。

#### 页面 4：可靠性行为

覆盖：

- 延迟请求。
- 失败请求。
- 下载。
- iframe。
- redirect。
- SPA URL 变化。

### 2.4 端到端检查链

```text
Playwright / Browser 自动操作测试页面
        ↓
Capture 被动观察
        ↓
检查 Raw events.jsonl
        ↓
检查 Body / Download 文件
        ↓
结束 Session
        ↓
检查 Finalization
        ↓
检查 AI-ready 索引和脱敏
```

注意：

> 测试程序可以操作测试站点；正式 Capture 自身仍不能主动操作真实页面。

## 3. 关键自动化验收用例

### 3.1 Request 完成晚于 Step

```text
Request starts in Step 1
→ Step 1 ends
→ Response completes later
```

要求：

- Request 仍属于 Step 1。
- 标记 completed_after_step。

### 3.2 Request 完成晚于 Session

```text
Request starts before Session end
→ Session end clicked
→ Response completes within 10s
```

要求：

- Request 仍属于 Session。
- 正常完成。

超过 10s：

- incomplete。

### 3.3 Session 开始前请求

```text
Request starts before Session
→ Response after Session start
```

要求：

- 整条排除。
- 不保存半条 Response。

### 3.4 Sidecar 刷新

要求：

- Session 不受影响。
- Step 不受影响。
- 重开后恢复状态。

### 3.5 Browser 关闭

要求：

- Session INTERRUPTED。
- reason 正确。
- 不可恢复 ACTIVE。

### 3.6 Engine stop

要求同上。

### 3.7 Interrupted 重启

要求：

- 重启后发现未处理 Session。
- 阻止新 Session。
- 允许封存或删除。

### 3.8 Raw 写失败

要求：

- 关键失败后立即中断。
- 不继续显示正常录制。

### 3.9 Body 超限

要求：

- Session 继续。
- 明确 gap。
- 完整性摘要出现。

### 3.10 AI-ready 失败

要求：

- Session 仍 SEALED。
- AI-ready FAILED。
- 可重新生成。

## 4. 真实听云人工验证计划

这部分只写计划，由用户亲自决定何时执行、怎么操作、是否达标。

### 场景 1：复杂表单与最终提交

#### 建议目标

选择一个真实配置页面，例如告警配置。

#### 建议操作

```text
打开表单
→ 查看实际选项
→ 切换上游选项
→ 观察下游级联
→ 填写阈值
→ 继续选择其他字段
→ 真实点击保存
```

#### 重点检查

- 实际暴露的选项是否留下。
- 显示文字和技术值是否正确。
- 级联前后状态是否能看懂。
- 提交前完整表单状态是否存在。
- 最终 Request Payload 是否完整。
- Response 是否完整。
- Review 是否清爽。
- AI-ready 是否足够外部 Agent 深挖。

### 场景 2：问题 → 事务 → Trace → URL

#### 建议目标

从一条真实告警或真实问题出发。

#### 建议操作

```text
进入应用
→ 找事务
→ 找 Trace
→ 打开具体详情
→ 允许新 Tab
→ 在关键时刻写备注
```

#### 重点检查

- 多 Tab 是否连续。
- Step 能否跨 Tab。
- URL / Query / Hash 是否完整。
- SPA URL 变化是否记录。
- 过程中出现的业务 ID 是否保留。
- Request / Response 是否足以让外部 AI 发现候选 Endpoint。

Capture 本身不判断哪个 URL 可放报告。

### 场景 3：长 Session 可靠性

#### 建议覆盖

- 多 Step。
- Step 之间停顿。
- 新 Tab。
- 请求完成晚于 Step。
- Session 结束时 pending request。
- Sidecar 刷新。
- 较多 Request / Response。

#### 重点检查

- Raw 是否持续增量落盘。
- Session 是否正常封存。
- 完整性摘要是否可信。
- AI-ready 是否生成。
- Review 是否没有明显卡顿或失控。

## 5. Fresh AI 阅读测试

### 5.1 目标

验证 AI-ready 是否真的机器友好。

### 5.2 方法

1. 选择一个真实 Session。
2. 只提供 AI-ready 包。
3. 不额外口头解释。
4. 让一个没有上下文的外部 Claude Code / Codex / Agent 先读 `README_FOR_AI.md`。

### 5.3 让外部 Agent 回答

至少包括：

1. 这次用户做了什么？
2. 探索路径是什么？
3. 哪些请求可能值得进一步研究？
4. 表单实际发生了哪些级联变化？
5. 最终提交 Request / Response 在哪里？
6. 哪些页面 URL 值得进一步验证？
7. 哪些结论无法确定？

### 5.4 验证重点

外部 Agent 应能使用稳定 ID 回引：

```text
interaction-...
form-state-...
request-...
response-...
```

如果仍需用户大量补充背景，说明 AI-ready 组织方式需要改进。

## 6. 不要求真实环境自动化

不做：

- CI 自动登录听云。
- CI 自动提交真实告警。
- Capture 自动遍历真实页面。
- 程序自动判断真实场景“通过 / 失败”。

真实环境是否满足需求，由用户最终判断。
