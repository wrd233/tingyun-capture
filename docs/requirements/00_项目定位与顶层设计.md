# 00 项目定位与顶层设计

## 1. 背景

听云 APM 当前存在两类互补事实来源：

1. 官方 API 文档给出“系统声明支持什么”。
2. 真实 UI 操作给出“系统实际上怎样工作”。

真实页面行为往往比官方 API 更复杂：

- 一个需求会跨多个 API 前缀。
- 页面操作和接口调用不是一一对应。
- 参数可能在多个请求之间流转。
- 新标签页会改变探索路径。
- 表单选项可能异步加载并产生级联。
- 最终保存请求往往依赖前置页面状态。
- Trace、事务、应用等页面中的 ID 和 URL 对后续报告回到问题现场很重要。

因此需要一个独立事实层，把“用户围绕某个需求在真实听云平台做了什么”完整保存下来。

## 2. 在整个听云工程中的位置

```text
官方 API 文档 + 真实 UI 行为
                  ↓
           tingyun-capture
                  ↓
    Raw Facts / Operation Process
                  ↓
       综合知识目录与能力理解
                  ↓
             tingyun-cli
                  ↓
        Agent / 自动化工作流
                  ↓
          报告 + 问题现场 URL
```

### tingyun-capture

职责：

- 记录真实探索过程。
- 保存浏览器层事实证据。
- 组织 Session / Step / Note。
- 生成脱敏、可引用、AI 友好的证据包。

不负责：

- 判断 Endpoint 是否重要。
- 推断参数血缘。
- 推断级联因果。
- 自动生成 Skill。
- 自动遍历页面。
- 执行 API。

### tingyun-cli

职责仍然是只读的“手”和事实读取层。它不是本轮改造对象。

### 外部 AI / Agent

读取 Capture 产物后负责：

- 发现候选 Endpoint。
- 理解参数来源。
- 提炼探索路径。
- 识别未来可自动化的网络链。
- 判断 URL 是否值得进一步验证。

## 3. 产品定义

### 3.1 核心定义

> `tingyun-capture` 是一个 **need-driven real-platform exploration recorder**。

一次 Session 的目标不是“抓尽可能多的包”，而是让另一个没有在场的 AI 能够理解：

1. 用户围绕什么主题进行探索。
2. 用户按什么顺序做了哪些关键操作。
3. 页面、Tab 和 URL 怎样变化。
4. 哪些网络请求与响应真实发生。
5. 哪些表单选项被实际暴露和选择。
6. 级联前后状态如何变化。
7. 最终真实提交时页面状态、Payload 和 Response 是什么。
8. 哪些事实缺失或未完整保存。

### 3.2 高保真的含义

本项目中的“高保真”指：

> 在浏览器调试层能够可靠观察的范围内，忠实保存真实用户操作和真实网络事实。

它不是：

- TCP 抓包器。
- TLS 原始流量采集器。
- Wireshark 替代品。
- DOM 录像系统。
- 浏览器 Replay 录制器。

## 4. 顶层原则

### 4.1 事实与推理分离

```text
Raw Facts
≠
AI Conclusions
```

Raw 只保存：

- 实际发生的时间。
- 实际操作。
- 实际页面状态。
- 实际 URL。
- 实际 Request / Response。
- 用户自己写的说明。

Capture 不把推理结果伪装成事实。

### 4.2 用户操作纯净性

Capture 永远不主动：

- 点击页面。
- 展开下拉框。
- 选择选项。
- 提交表单。
- 穷举组合。
- 为发现更多接口而制造请求。

因此：

> Capture 记录到的页面操作，必须等于用户真实做过的操作。

### 4.3 Session 是连续事实边界，Step 是语义标记

```text
Session：持续高保真事实边界
Step：用户主动标记的一段操作意图
```

Step 不控制采集开关。

Step 之间仍持续记录事实；内部可以存在无 Step 区间，但用户无需理解这个概念。

### 4.4 Raw 是唯一真相源

```text
Raw Evidence
→ 唯一不可替代事实源

Review Index / AI-ready
→ 派生产物，可重建
```

### 4.5 重要内容优先

项目必须避免“能抓什么就把什么做成产品能力”。

真正优先关注：

- Endpoint。
- Request / Response。
- 参数与 Body。
- 页面操作上下文。
- 表单选项与级联。
- 最终提交。
- Trace / 事务 / 应用等探索路径。
- URL 变化与问题现场回达。

非核心细节只能在“底层直接可得、几乎无额外复杂度”时附带保留，且不应污染 UI。

## 5. 用户与环境

### 5.1 用户

v1 面向项目维护者本人，不考虑普通终端用户。

### 5.2 平台

仅正式支持：

```text
macOS
```

Windows / Linux：

- 不承诺。
- 不测试。
- 不为其增加专门适配复杂度。

### 5.3 产品形态

本地开发者工具：

```text
一条命令启动
├── Capture Engine
├── Sidecar Web UI
└── Dedicated Chromium
```

不做：

- macOS `.app`。
- DMG。
- 菜单栏 App。
- 自动更新器。
- 复杂安装器。

### 5.4 本地网络边界

所有本地服务仅监听：

```text
127.0.0.1
```

不支持：

- 局域网访问。
- 远程访问。
- 登录系统。
- 用户权限系统。
- 本地 HTTPS。

## 6. 成功标准

v1 成功不是“功能很多”，而是以下链条真正成立：

```text
用户在听云中真实探索
        ↓
Capture 无感、连续、可靠记录
        ↓
用户能在清爽 Review 中确认核心事实
        ↓
生成脱敏 AI-ready 包
        ↓
一个没有额外上下文的外部 Agent
能够理解路径并回引具体 evidence ID
```
